# Evoluzioni DevOps / infra (documento vivo)

Registro delle scelte fatte in versione **cost-min / semplice** ora, con il **percorso di hardening/scaling**
da attivare quando i requisiti lo giustificano. Aggiornato a ogni decisione rilevante. Ogni voce: stato attuale →
evoluzione, **trigger** (quando farlo), costo/impatto.

| # | Tema | Stato attuale (PoC) | Evoluzione | Trigger | Costo evoluzione |
|---|---|---|---|---|---|
| E1 | **Uscita rete servizi (NAT)** | Subnet **pubbliche** + security group stretti, **niente NAT** (#06 B) | Subnet **private** + **NAT Gateway** | Requisiti di isolamento/compliance, o tasks che non devono avere IP pubblico | ~$32/mese/ambiente |
| E2 | **Ingress API→container** | **VPC Link + Cloud Map** (service discovery), no ALB (#06 B) | **Application Load Balancer** | Traffico in crescita, health check avanzati, routing/sticky, WAF su ALB | ~$16/mese/ambiente + LCU |
| E3 | **Alta disponibilità Fargate** | **1 task** per servizio (no HA) | ≥2 task multi-AZ + autoscaling | Carico reale / SLA di uptime | costo ∝ n. task |
| E4 | **Aurora prod** | **scale-to-0 anche su prod** (#06 E) | min ACU >0 su prod per latenza costante | Cold-start del DB inaccettabile in prod | da ~$44/mese (0.5 ACU on) |
| E5 | **Ambienti effimeri PR** | Solo local/test/prod (#12) | Env effimero per PR in CI | Team più grande / molte PR parallele | variabile |
| E6 | **WAF** | Nessun WAF (#06 F) | AWS WAF su CloudFront/API GW | Esposizione pubblica con traffico reale / requisiti di sicurezza | ~$5–10/mese + per-regola/richiesta |
| E7 | **Cognito Advanced Security** | Lockout built-in + throttling API GW (usecases/01) | Cognito Advanced Security (adaptive, credenziali compromesse) | Traffico reale / requisiti anti-frode | a pagamento per MAU |
| E8 | **Runner CI build native** | GitHub **hosted runner** standard (2 vCPU/7GB), native solo on-demand `[graal]` (#07 D) | Runner più grande / **self-hosted** + caching aggressivo | Build native GraalVM troppo lente o minuti free tier esauriti | minuti a pagamento o costo runner self-hosted |
| E9 | **Promozione immagini servizi** | Gate prod: solo native già su ECR per lo SHA; recupero via toggle manuale (#07 H) | Build native automatica a ogni merge (parità totale, zero gate) | Rilasci frequenti / dimenticanze `[graal]` ricorrenti | +minuti CI |
| E10 | **Backend tracce (tracing)** | App **strumentata OTel** + trace context propagato, **export SPENTO**, nessun collector (#08 C, c1) | **Accendi le tracce**: collector **ADOT → X-Ray** (c2), oppure **OTel Collector → Grafana Cloud/Tempo** | Debug di latenza/flussi complessi sotto carico reale | X-Ray ~$5/1M tracce + collector (sidecar RAM o ~$10/mese centrale), o Grafana Cloud |
| E11 | **Stack observability completo OTel+Grafana** | Backend AWS-native (CloudWatch Logs/Metrics) + strumentazione OTel/Micrometer neutra (#08 A/B) | Spostare il **backend** su Grafana (Loki/Tempo/Mimir o **Grafana Cloud**) ripuntando gli exporter | Si vuole UI/correlazione Grafana; team/volumi maggiori | Grafana Cloud (free/pro) o stack self-hosted (oneroso) |
| E12 | **Error tracking dedicato** | CloudWatch-native: ERROR JSON + metric filter + allarme + triage Logs Insights (#08 F, f1) | **Sentry** (SaaS, free tier) / **GlitchTip** (self-host) / error-tracking Grafana: dedup, release tracking, breadcrumb | Triage "a query" troppo scomoda; serve grouping/UX migliore (specie frontend, #08 H) | Sentry free/pro o costo self-host; valutare uscita dati per GDPR |
| E13 | **Uptime monitor esterno (non-AWS)** | Canary **EventBridge+Lambda in eu-central-1** (cross-region, ~$0, #08 G) | Monitor indipendente fuori da AWS: **UptimeRobot** (free) / **CloudWatch Synthetics** | Serve vantage point per outage AWS multi-regione / SLA serio | UptimeRobot free o Synthetics ~$5/mese/canary |
| E14 | **RUM frontend completo** | Cattura **errori JS only** → CloudWatch, niente tracking comportamentale (#08 H, h1) | **CloudWatch RUM** / **Sentry** / **Grafana Faro**: performance, Core Web Vitals, sessioni | Serve performance percepita/sessioni; **consent-gated** (#13) | CloudWatch RUM ~$1/100k eventi, o SaaS free/pro + gestione consenso |
| E15 | **Monitoraggio per-tenant "VIP" / SLA per cliente** | Regola dei due piani: `tenant_id` solo nei log/tracce, metriche/allarmi per-app, non per-tenant (#08 J) | Metriche/allarmi/dashboard **dedicati per tenant VIP** (SLA per cliente), gestendo la cardinalità (whitelist tenant, non tutti) | Clienti enterprise con SLA contrattuale / esigenza di report per-tenant | costo ∝ n. tenant monitorati (cardinalità) |
| E16 | **Load/performance testing** | Solo misura **cold-start** documentata; no load test (scale-to-0/1-task non rappresentativo, #10 K) | **k6** (script JS) per load/stress + SLA di latenza/capacità | Avvicinamento al go-live / requisiti di SLA / scaling reale | minuti CI/risorse di test |
| E17 | **Visual regression "AI"** | aria/DOM snapshot + pixel tollerante, baseline gestite dall'agente, no approvazioni manuali (#10 F) | **Applitools/Percy/Chromatic** (visual AI + UI di approvazione) | Servono diff visivi gestiti/branded e workflow di review dedicato | SaaS a pagamento (coinvolge l'utente nell'approvazione) |
| E18 | **E2E multi-browser** | Playwright **Chromium-first** (#10 F) | Aggiungere **WebKit/Firefox** | Garanzia cross-browser su traffico reale | +minuti CI |
| E19 | **Contract testing consumer-driven** | OpenAPI fonte di verità + tsc su client generato + oasdiff bloccante (#10 G) | **Pact** (consumer-driven) | FE/BE diventano team/repo separati con evoluzione indipendente | setup Pact broker/CI |
| E20 | **Infra testing reale** | `plan` + `terraform test` (no risorse reali, #10 H) | **Terratest** (crea risorse reali e asserisce) | Moduli infra complessi / serve verifica end-to-end dell'apply | costo risorse temporanee + minuti |

> Nota: E1–E7 = **modifiche Terraform**; E8–E15 = **CI/observability** (#07/#08); E16–E20 = **testing** (#10). App **non**
> va re-instrumentata (strumentazione neutra OTel/Micrometer): le evoluzioni observability sono cambi di **backend/config**,
> non di codice. Principio invariato: isolamento incrementale.

| E21 | **Rotation credenziali DB per-app** | Password generata da Terraform in Secrets Manager, riallineabile ri-applicando (la Lambda `db-bootstrap` è idempotente); nessuna rotation automatica (UC 0004) | **Rotation automatica** Secrets Manager (Lambda di rotation dedicata) + restart/refresh dei task | Requisiti di sicurezza/compliance su rotazione periodica delle credenziali | Lambda rotation (~$0) + gestione grazia doppia-password |
